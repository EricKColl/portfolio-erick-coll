PRODUCTO 2 — SEGUNDA INICIALIZACIÓN

El Producto 1 fue la demostración de que un equipo puede construir la capa visible. El Producto 2 es la demostración de que ese mismo equipo sabe hacer que esa capa funcione de verdad — con APIs nativas del navegador, persistencia real, interacciones sofisticadas y código organizado con la madurez de alguien que ya no improvisa.

localStorage para sesión. IndexedDB para datos de empleos. Canvas para gráficos nativos. Drag & Drop para el dashboard. Promesas y async/await para todas las operaciones asíncronas. Módulos ES6 para que el código tenga fronteras, no spaghetti. Vanilla JS puro, porque si controlas el estándar, controlas todo lo demás.

API

HTML5 APIs — EL ARSENAL NATIVO

El navegador moderno es una plataforma de ejecución completa. El Producto 2 explota cuatro de sus APIs más potentes para convertir JobConnect en una aplicación que no solo parece funcional sino que lo es de verdad. Sin jQuery. Sin frameworks. Solo el estándar, bien usado.

4 APIs ACTIVAS
CVS
Canvas API
Gráfico nativo de ofertas/demandas en la página de empleos. Dibujado línea a línea con la Context 2D API: barras, etiquetas, colores diferenciados por tipo. Sin Chart.js ni librerías externas — solo ctx.fillRect() y determinación.
int_3_empleos.js
WS
Web Storage API
localStorage almacena el email del usuario activo para que la barra de navegación muestre quién está logueado en todas las páginas. Persistente entre sesiones, sincrónico y simple. El CRUD de usuarios vive en localStorage.
almacenaje.js
IDB
IndexedDB
Base de datos NoSQL en el navegador para almacenar ofertas y demandas de empleo. Object stores, índices, transacciones, cursores y promesas que envuelven la API de bajo nivel. Todo sin servidor. Todo sin conexión. Todo sin excusas.
almacenaje.js · IDB
D&D
Drag & Drop API
El dashboard genera tarjetas arrastrables de empleos. El usuario las arrastra al contenedor de selección. Los eventos dragstart, dragover y drop gestionan el flujo completo. Los datos viajan en dataTransfer. Interacción natural, código limpio.
int_1_dashboard.js
jobconnect@navegador — almacenaje.js — IndexedDB + localStorage
¿POR QUÉ IndexedDB PARA EMPLEOS? localStorage tiene límite de ~5MB y solo acepta strings. IndexedDB maneja objetos complejos, soporta índices para consultas rápidas y funciona de forma asíncrona sin bloquear el hilo principal. Para una app de empleo real, no hay otra opción.
¿POR QUÉ Canvas NATIVO SIN LIBRERÍAS? Porque el objetivo de la asignatura es demostrar conocimiento del estándar. Chart.js lo haría en 3 líneas, pero no te enseña cómo funciona la Context 2D API. Aquí cada barra, cada eje y cada etiqueta es código deliberado, no delegado.
MOD

ARQUITECTURA MODULAR — ES6 MODULES

El código del Producto 2 no es un archivo de 800 líneas con todo mezclado como un cajón de sastre digital. Es un sistema modular donde cada archivo tiene una responsabilidad clara y todas las páginas hablan con el mismo módulo central para las operaciones de datos.

MODULAR · ASYNC · ES6
almacenaje.js — EL NÚCLEO Módulo central con todas las operaciones CRUD. Es el único punto de contacto con localStorage e IndexedDB. Cualquier página que necesite datos lo importa. Cambia este módulo en el Producto 4 para sustituir fetch() y el resto del frontend no se entera.
int_1_dashboard.js Recupera empleos de IndexedDB, genera tarjetas dinámicamente y las convierte en draggables. Gestiona los eventos dragstart, dragover y drop. Muestra el usuario activo desde localStorage. La lógica de interacción del panel principal.
int_3_empleos.js Alta, consulta y borrado de ofertas/demandas con IndexedDB. Genera la tabla dinámica de empleos y llama a la función que dibuja el gráfico Canvas. Maneja el formulario, valida campos y actualiza la vista tras cada operación.
int_4_usuarios.js + login.js Gestión de usuarios con localStorage: alta con validación de email duplicado, listado dinámico y borrado. El login verifica credenciales y guarda el usuario activo. Ambos importan almacenaje.js y no tocan el almacenamiento directamente.
PROMESAS Y ASYNC/AWAIT Toda la interacción con IndexedDB es asíncrona por naturaleza. Las operaciones están envueltas en promesas que se resuelven cuando la transacción completa. Los módulos de cada página usan async/await para que el código asíncrono sea tan legible como el síncrono. Porque await hace que el código asíncrono parezca honesto.
int_3_empleos.js — async/await + IndexedDB + Canvas

LAS 4 PANTALLAS — FUNCIONALIDAD DETALLADA

Cada página del sistema tiene su función, su módulo y su relación con las APIs. Así funciona cada una por dentro.

4 PANTALLAS OPERATIVAS
DASH
Dashboard (int_1_dashboard.js)
Recupera empleos de IndexedDB al cargar. Genera tarjetas arrastables dinámicamente. Los eventos dragstart/dragover/drop mueven ofertas/demandas al contenedor de selección. Los datos del elemento arrastrado viajan en dataTransfer. Muestra usuario activo desde localStorage. Dos botones de filtro (Oferta/Demanda) opcionales para exploración avanzada.
IndexedDB + Drag&Drop + localStorage
LOGIN
Login (login.js)
Formulario de email + contraseña. Al enviar, llama a almacenaje.loguearUsuario() que verifica credenciales contra localStorage. Si el login es exitoso, guarda el email del usuario activo en localStorage y muestra confirmación. mostrarUsuarioActivo() lee ese dato en todas las páginas para mostrar quién está conectado en la navbar.
localStorage · DOMContentLoaded
EMP
Empleos (int_3_empleos.js)
CRUD completo sobre IndexedDB: alta con título/email/fecha/descripción/tipo, tabla dinámica con botón de borrado por fila, y un gráfico Canvas nativo que visualiza el conteo de Ofertas vs Demandas. El canvas se actualiza automáticamente tras cada operación. Sin librerías de gráficos externas. La barra más honesta del sistema.
IndexedDB + Canvas API
USR
Usuarios (int_4_usuarios.js)
Alta de usuarios con nombre/email/contraseña validados. Verificación de email duplicado antes de persistir. Tabla dinámica de todos los usuarios con borrado por fila. Todo almacenado en localStorage como array JSON. No es multiusuario — es frontend de gestión local, que es exactamente lo que el Producto 2 pide.
localStorage · validaciones

MAPA TEMPORAL — EVOLUCIÓN DEL SISTEMA

Las decisiones que llevaron al Producto 2 desde la base del Producto 1 hasta el sistema con persistencia, interacción y gráficos que ahora existe.

6 FASES
01 · PUNTO DE PARTIDA — PRODUCTO 1 Se parte del frontend del P1: dashboard, login, empleos y usuarios con datos en arrays de memoria. La arquitectura modular del P1 facilita la evolución. El código ya tiene fronteras — solo hay que profundizar lo que hay detrás de ellas.
02 · DISEÑO DE LA CAPA DE PERSISTENCIA Se decide qué va a localStorage (usuarios, sesión activa) y qué va a IndexedDB (empleos, por volumen y complejidad). Se diseña el módulo almacenaje.js como la única interfaz entre las páginas y el almacenamiento. Una decisión que el Producto 4 agradece.
03 · IMPLEMENTACIÓN DE IndexedDB + localStorage Se construye almacenaje.js con todas las funciones CRUD: abrirBD, agregarEmpleo, obtenerEmpleos, borrarEmpleo para IndexedDB; y alta/baja/login para usuarios en localStorage. Todo envuelto en promesas, todo consumible con async/await.
04 · DASHBOARD CON DRAG & DROP Se implementa la generación dinámica de tarjetas de empleos y los eventos HTML5 Drag & Drop. Cada tarjeta tiene draggable="true" y gestiona dragstart. El contenedor destino gestiona dragover y drop. Los datos del empleo viajan en dataTransfer.setData().
05 · CANVAS NATIVO EN EMPLEOS Se implementa el gráfico de empleos con Canvas 2D API pura. Fondo, ejes, barras, etiquetas y leyenda — todo dibujado con ctx.fillRect(), ctx.fillText() y ctx.strokeRect(). El gráfico se regenera cada vez que cambian los datos. Primitivo. Potente. Propio.
06 · INTEGRACIÓN, TESTING Y DOCUMENTACIÓN Se conectan todos los módulos, se verifica el flujo completo del sistema (login → dashboard → empleos → usuarios), se generan los vídeos teórico y práctico, se redacta la documentación técnica y se prepara este panel de presentación. El Producto 3 tiene frontend esperando.

SECUENCIA COMPLETADA — PRODUCTO 2 ONLINE

Con este producto, el frontend de JobConnect ha cruzado el umbral entre "interfaz con buena pinta" y "aplicación que funciona de verdad cuando nadie mira". Los datos persisten. Los empleos viven en IndexedDB. Los usuarios en localStorage. El gráfico dibuja lo que hay. El dashboard deja arrastrar lo que importa.

No hay backend todavía — y sin embargo el frontend no lo echa de menos porque sabe exactamente qué habrá que sustituir cuando llegue. El módulo almacenaje.js es el único punto de cambio. Eso es arquitectura limpia. Eso es el Producto 2. Lo que viene en el 3 ya tiene frontend esperando.
