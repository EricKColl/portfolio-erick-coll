NEXUS // BACKEND REAL OPERATIVO

El Producto 3 transforma la lógica de almacenamiento del Producto 2 en un servicio backend real. La aplicación abandona las estructuras del navegador y pasa a gestionar sus datos desde una API GraphQL conectada a MongoDB. Sin frontend en este producto: la verificación se hace con Postman y Apollo Sandbox, donde el sistema demuestra que lo que ahora persiste es real.

visión general
01 OBJETIVO TÉCNICO Migrar la lógica de datos hacia un backend Node.js + Express, exponiendo operaciones mediante GraphQL y validando el sistema sin frontend.
02 MODELO DE API GraphQL permite trabajar con un único endpoint y solicitar exactamente los campos necesarios en cada operación, eliminando overhead.
03 PERSISTENCIA REAL MongoDB almacena documentos JSON en colecciones reales, sustituyendo las estructuras en memoria del Producto 2.
04 VALIDACIÓN PROFESIONAL Postman y Apollo Sandbox permiten comprobar queries, mutations, autenticación, errores y pruebas automatizadas sin frontend.

ARQUITECTURA MODULAR DEL BACKEND

El backend se organiza por capas para separar responsabilidades: configuración, schema GraphQL, resolvers, modelos, middleware, utilidades, seed inicial y conexión con MongoDB. Modularidad real que el Producto 4 agradece.

modularidad
⚙src/configVariables de entorno, conexión a MongoDB y configuración centralizada del runtime.
⊕src/graphqlSchema, types, queries, mutations y resolvers que definen el contrato completo de la API.
◇src/modelsAcceso directo a colecciones MongoDB sin Mongoose. Lógica de negocio encapsulada por entidad.
⚿src/middlewareJWT auth, manejo de errores, contexto de autorización entre cliente y resolvers.
⌬src/utilsValidators, errores tipados y helpers que mantienen el código limpio sin duplicación.
⌖src/seedInicialización del administrador y datos base. El sistema arranca útil desde el primer commit.
GQL

API GRAPHQL // ENDPOINT ÚNICO

GraphQL convierte la API en un contrato tipado. Un solo endpoint, schema explícito, queries que piden exactamente lo necesario y mutations que documentan cada operación de escritura. Sin sobrecarga de datos. Sin múltiples endpoints. Sin sorpresas.

schema first
QQUERIESlistarUsuarios, listarPublicaciones, listarSeleccionadas, resumenDashboard, healthcheck del sistema.
MMUTATIONScrearUsuario, eliminarUsuario, crearPublicacion, eliminarPublicacion, gestionar seleccionadas, loginAdmin.
RRESOLVERSConectan cada operación con los models. La API no toca persistencia directamente — separación de capas real.
TTYPEDEFSDefine types, inputs, argumentos, respuestas y directivas. Contrato auto-documentado y verificable en Apollo Sandbox.

JWT

SEGURIDAD // AUTH + CONTROL

El sistema incorpora autenticación JWT para proteger operaciones administrativas. bcryptjs hashea contraseñas, el middleware extrae el token Bearer del header, los resolvers verifican el contexto y los errores son tipados. Una API que no se confía de quien le habla.

api protegida
→loginAdminMutation pública para autenticar al administrador y emitir el token JWT firmado.
⌗JWTToken firmado HS256 con expiración. Identifica al usuario sin sesiones de servidor.
BBearer TokenCabecera Authorization usada en operaciones protegidas y verificada por middleware.
#bcryptjsHash de password con salt antes de persistir. Texto plano nunca toca la base de datos.
! ERRORES TIPADOS Unauthorized · Forbidden · ValidationError · ConflictError · NotFoundError. Respuestas coherentes y trazables que el cliente puede manejar de forma predecible.
★ PROTECCIÓN DE DATOS El campo password nunca se expone en respuestas GraphQL. Los resolvers filtran campos sensibles por defecto. La seguridad es un default, no un opt-in.

VALIDACIÓN // POSTMAN + APOLLO

Sin frontend, la validación se hace donde toca: Postman ejecuta la batería de pruebas funcionales y de seguridad, Apollo Sandbox permite explorar el schema. 22 requests organizadas, environments configurables y tests automáticos que verifican comportamiento. El sistema se sostiene solo.

testing automatizado

⊕CRUDPruebas sobre usuarios, publicaciones, dashboard y seleccionadas con datos reales.
✗TESTS NEGATIVOSValidan que una mutación sin token o un login incorrecto no se acepten. Negar también es validar.

DESPLIEGUE CLOUD // ATLAS + SANDBOX

El backend queda preparado para ser probado fuera del entorno local, conectando el código ejecutado en CodeSandbox con una base de datos real alojada en MongoDB Atlas. Variables de entorno, conexión segura, endpoint público y verificación externa: cloud-ready, sin trampa.

cloud ready
01 · MONGODB ATLAS Base de datos cloud con colecciones reales, documentos persistentes e índices definidos para optimizar consultas y garantizar coherencia entre operaciones simultáneas.
02 · CODESANDBOX Entorno cloud donde el backend se ejecuta con Node.js, dependencias instaladas y endpoint GraphQL accesible externamente. Listo para que el evaluador lo abra y pruebe.
03 · VARIABLES DE ENTORNO Configuración segura de MONGO_URI, MONGO_DB_NAME, JWT_SECRET, ADMIN_EMAIL y ADMIN_PASSWORD sin exponer credenciales en GitHub. .env.example documenta sin filtrar.
04 · VERIFICACIÓN EXTERNA El sistema puede probarse desde Apollo Sandbox y Postman usando el endpoint público desplegado. La distancia entre local y producción se vuelve invisible.
